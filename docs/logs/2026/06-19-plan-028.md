## 关键发现

**根因**：`/api/teams` handler（`src/app.py:166-183`）用 **API name_en** `"Bosnia and Herzegovina"` 去查 `countries.json`，但 `countries.json` 里的 key 是 **ICS 拼写** `"Bosnia & Herzegovina"`——查不到 → `name_zh=None`。

```python
# src/app.py:173
meta = lookup_country(t.get("name_en") or "")  # ← "Bosnia and Herzegovina"
enriched[tid] = {
    "name": t.get("name_en") or tid,
    "name_zh": (meta or {}).get("name_zh"),   # ← None
    ...
}
```

**深层根因（同 Plan 027）**：两套数据源（baires ICS + worldcup26.ir API）拼写不一致。Plan 027 修了"积分计算"链路（`build_team_id_map`），但**没修"中文名查表"链路**（`lookup_country` → `/api/teams` handler）。

**踩坑名单**（同 Plan 027 一致）：

| 队 | API name_en | countries.json key | 之前缺中文？ |
|---|---|---|---|
| 波黑 (B 组) | `Bosnia and Herzegovina` | `Bosnia & Herzegovina` | ✅ |
| 美国 (D 组) | `United States` | `USA` | ✅ |
| 民主刚果 (K 组) | `Democratic Republic of the Congo` | `DR Congo` | ✅ |

## 决策日志

### 1. lookup 加 fallback vs handler 改用 build_team_id_map
**问题**：3 支队的跨源名字不一致怎么修？
**候选**：
- (A) `lookup_country` 加 3-pass fallback（**采用**）
- (B) `/api/teams` handler 改用 `build_team_id_map` 思路
- (C) 改 `countries.json` 增补别名
- (D) 改 ICS parser
**决定**：A
**理由**：
- B 把 resolver 泄漏到调用方
- C 改手维护数据（`data/countries.json` 是 L1）缩放差
- D 改真相源侵入大
- A 单一职责：lookup 是入口，逻辑收敛

### 2. _norm_team_key 提到 countries.py 还是 details.py
**问题**：Plan 027 在 details.py 定义了 `_norm_team_key`。Plan 028 也要用。
**决定**：提到 `countries.py` 公开
**理由**：
- 避免循环 import（details 依赖 countries，不能反向）
- 单一真相源：两边用同一个函数
- Plan 027 测试不破（re-export `_norm_team_key` 别名）

### 3. 缩写差异（USA/DRC）怎么修
**问题**：Pass 2 normalize 救不了缩写。Pass 3 直接比对 code_fifa 也不通（"United States" ≠ "USA"）。
**决定**：加显式 alias 表 `_TEAM_NAME_ALIASES`
**理由**：
- 显式 > 隐式：3 支队是已知 case，写明清清楚楚
- 缩放足够：48 队里只有 2-3 个会踩
- 易扩展：未来新 case 加一行即可
- 试过 API bridge 方案（`lookup()` 接受 `teams_by_id` 参数），但会引入 countries → worldcup_api 循环依赖，放弃

### 4-pass lookup 行为

| Pass | 触发条件 | 覆盖 case |
|------|---------|-----------|
| 1. 精确 | `name_en == countries.json key` | 45/48 队（所有拼写一致的）|
| 2. normalize | `norm(name_en) == norm(countries.json key)` | Bosnia (`and` ↔ `&`) |
| 3. 显式 alias | `name_en in _TEAM_NAME_ALIASES` | USA, DR Congo |
| 4. code 反查 | `name_en.upper() == code_fifa/code_iso` | "USA" / "US" 直接传 code（边界）|

### 关键收获

- **跨源拼写不一致有 3 种形态**：
  1. 字符差异（`&` vs `and`）→ normalize 解决
  2. 缩写差异（`USA` ↔ `United States`）→ 显式 alias 表
  3. （无第 3 种；这次 Plan 028 把已知的全 cover 了）
- **lookup 是入口 → 把 fallback 收在 lookup 内部**，调用方无感
- **`_norm_team_key` 提到 countries.py** 是合理的重构，**避免循环 import** + 单一真相源
- **不再做**：API bridge 方案（dependency injection 太重）、改 countries.json 增补字段（schema 变更太重）
