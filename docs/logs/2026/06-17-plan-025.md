## 决策记录

### 1. Root cause 升级

- **v1 假设**：手维护数据把 `90+6'` 塞进 player 字段
- **v2 真相**：`src/data/worldcup_api.py:188` 的 `_parse_scorer_strings` 正则只认 `90'+6'`（`'` 在 `+` 前），worldcup26.ir API 实际返回 `90+6'`（`'` 在后）。每次 `/api/refresh` 都重新生成 malformed entry，Plan 017.1 的 `_is_incomplete` → `merge_from_api` 链路被上游 bug 绕过（API 自身就是错的，覆盖后还是错的）
- **决定**：v2 升级 plan scope，从「手动修数据」变成「修 parser + 修数据 + 兜底 standings」

### 2. parser 正则双向支持

旧：`r"(?:'(\+(\d{1,3}))?)?"` —— 强制要求 `+N` 前有 `'`
新：`r"(?:'?\+(\d{1,3})'?)?"` —— `'` 完全可选，两侧任意

顺手也改了 suffix 组：`r"(?:'?\(([^)]+)\))?"` 让 `'` 可以被 suffix 消费（own goal 格式 `7'(OG)` 仍工作）

group 编号变了：stoppage g4→g3，suffix g5→g4。代码里同步更新。

### 3. 本地 standings 兜底

worldcup26.ir `/get/groups` 在 Iraq-Norway 比赛结束后数小时仍显示 0 PTS（API 不更新）。**走本地推导**：
- 新增 `src/data/details.py:compute_standings_from_details()`
- 在 `src/app.py` 加 `_local_or_api_standings()` wrapper：先本地，失败/无数据时 fallback 到 API
- 优点：永远跟 goalscorers 同步，弹窗内数据一致；API 修好之后可以无成本切回去

### 4. 范围扩展

用户原话：「不只这两条数据」「积分榜也有问题」→ plan v2 把 5 项 fix 全部纳入：parser、测试、数据、refresh、standings 兜底

## 风险/未决

- **世界波 API 行为**：Iraq-Norway 比赛结束后数小时 API 仍未更新 Group I standings（手动刷新 / clear cache 后仍然 stale）。可能他们的 standings 计算有 bug 或延迟较长。本地兜底覆盖了，但若 worldcup26.ir 永远不更新，依赖 API 的其他视图（如赛事轮次的小组进度）也可能有问题 → 后续观察
- **Group A 等其他组**：本次只验证了 Group I。其他组同理：API 一旦 stale，本地立即接管，无需手动操作
- **球队名映射**：当前用 `name_en` 和 `fifa_code` 双向映射（`"Iraq" → "35"`）。若 API 增加新球队名格式，本地映射会找不到 → 退到 API fallback（`team_name_to_id.get(...) is None` 跳过该场）
