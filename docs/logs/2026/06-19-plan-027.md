## 关键发现

**直接根因**：`name_to_id` 字典只字面匹配 `name_en`/`fifa_code`，跨数据源字符差异即不匹配。

**字面对比**：

| ICS (baires) | API (worldcup26.ir) | 差异类型 |
|--------------|---------------------|---------|
| `Bosnia & Herzegovina` | `Bosnia and Herzegovina` | 字符 `&` vs `and` |
| `DR Congo` | `Democratic Republic of the Congo` | **缩写**（不可由 normalize 推导）|
| `USA` | `United States` | 缩写（已被 fifa_code 通道覆盖，不踩坑）|

**关键代码**（`src/app.py` 修复前）：

```python
name_to_id: dict[str, str] = {}
for tid, t in teams.items():
    for k in (t.get("name_en"), t.get("fifa_code")):
        if k:
            name_to_id[str(k)] = str(tid)
```

下游 `compute_standings_from_details`（`src/data/details.py`）拿不到 `home_id` 就 `continue` 整场被跳过。

**踩坑盘点**：
- B 组 6/12 加拿大 1-1 波黑 ❌（漏算 → 加拿大少 1 平、波黑整队缺）

## 决策日志

### 1. 多通道 resolver vs 改 ICS parser
**问题**：根因是名字不一致，怎么修？
**候选**：
- (A) 改 ICS parser 把名字统一成 API 拼写（`&`→`and`、`USA`→`United States`）
- (B) 新增多通道 resolver helper（**采用**）
- (C) 仅在 `app.py` 拼 `name_to_id` 时多查几个字段
**决定**：B
**理由**：
- A 改 baires 真相源（侵入大，未来 API 又改拼写还会再踩）
- C 把 resolver 泄漏到调用方
- B 解决根因（缺归一化层），自动覆盖未来同类 bug，单元测试容易写

### 2. resolver 用 5 个 pass 而不是 3 个
**问题**：原计划只写 3 个 pass（name_en / fifa_code / iso2 + normalize）。但 K 组测试发现 `DR Congo` vs `Democratic Republic of the Congo` 不可由字符 normalize 推导。
**决定**：加第 5 pass：countries.json 反查（`code_fifa`/`code_iso` 桥接）
**理由**：
- `countries.json` 已经存在并由用户手维护
- 5 pass 设计：API name_en → API fifa_code → API iso2 → normalize → countries.json
- 缩写类（`DR` ↔ `Democratic Republic`）必须靠外部映射，无法 derive
- 加完 5 pass 后 K 组也通过

### 3. helper 拆分 `_norm_team_key` + `build_team_id_map`
**问题**：单函数还是拆 helper？
**决定**：拆
**理由**：
- 单一职责：normalize 是字符串处理，build 是字典构建
- 单元测试可独立测（`TestNormTeamKey` 5 个 + `TestBuildTeamIdMap` 8 个）
- 未来再加新通道只动 `build_team_id_map`，不动 `_norm_team_key`
