## 决策日志

### 1. 不完整判定用"goalscorers 数量 < 比分总和"
**问题**: 怎么判断手维护的 entry 是"不完整"？
**候选**:
- (A) `len(goalscorers) < score.home + score.away`（采用）
- (B) `len(goalscorers) < 某个阈值`（5 个以上就算完整）—— 误伤 0-0 平局的没记录 case
- (C) 总是用 API 覆盖 —— 误伤手动修正（比如 965e9ac9ce78 的 Larin 78' 修正）
**决定**: A
**理由**: 简单、可解释、容易测试；不完整 = 跟比分数字对不上，必然漏写

### 2. 完整 entry 仍然胜过 API（不破坏手动修正）
**问题**: 现有 `merge_from_api` 是"existing always wins"，新规则会不会破坏用户的手动修正？
**决定**: 完整 entry 仍然胜出
**保护**:
- Plan 016 的 Larin 78' 修正（`965e9ac9ce78`）—— 2-0 比分有 2 个 goalscorers，完整，胜出
- `fdf1bfc654f3` (USA 4-1 Paraguay) —— 5 个 goalscorers 覆盖 4+1=5，完整，胜出
- `11a1e8291690` (Qatar 1-1 Switzerland) —— 2 个 goalscorers 覆盖 1+1=2，完整，胜出
- `744bfbfd7938` (Germany 7-1 Curacao) —— 8 个 goalscorers 覆盖 7+1=8，完整，胜出

跑了一遍 `pytest tests/`，所有现有 entry 都满足"完整"，没有任何被 API 覆盖的风险。

### 3. 改 `merge_from_api` 签名（返回 3-tuple）
**问题**: 老的 `(merged, added)` 不够区分"新增"vs"覆盖"
**决定**: 改成 `(merged, changed, overwritten)`，`changed = added + overwritten`
**影响**:
- `app.py:118` 改 1 行适配：`merged, scores_updated, _ = merge_from_api(...)`
- `test_worldcup_api.py` 2 个老测试更新签名
- `scores_updated` 字段对前端语义不变（仍然是"有多少 entry 被改动"）

### 4. 加 `_source` 字段记录手动维护的来源
**问题**: AGENTS.md 红线"不得在 data/*.json 手改后不说明来源"
**决定**: 瑞典-突尼斯这条加了 `_source: "新华社/央视 2026-06-15 12:00-12:15 报道..."`
**原因**: 这是 hand-maintained 字段（不是 schema 字段），下次 review 容易追溯

### 5. 不加 `_lock: true` 手动锁定
**问题**: 万一以后真的想保留某个 incomplete 条目怎么办？
**决定**: 不加
**理由**: 过度设计；当前需求"不完整=手维护出错"假设是合理的；如果真有反例，用户直接编辑 details.json 就好（YAGNI）

## Plan 017.1 Follow-up（13:00 完成）

### 触发

13:00 用户反馈德国 7-1 库拉索详情页 "K. Havertz 45'+5'(p)" 显示 0'。**Plan 017 的"数量不完整"检测抓不到这个** —— 德国 8 球对 7+1=8，数量对，但**单条数据格式坏了**。

### 改动

1. `src/data/details.py` `_is_incomplete` 加新规则：**任何 minute=0 都视为格式错误**（实际比赛进球至少 1 分钟）
2. `data/details.json` 修 2 处手维护错误：
   - 德国 vs 库拉索 Havertz 拆出 `minute=45, stoppage=5, type="penalty"`
   - 卡塔尔 vs 瑞士 Embolo 拆出 `minute=17, type="penalty"`
3. `tests/test_details.py` +2 测试
4. 200/200 单元测试全过

### 决策

- **用 "minute=0" 作为格式错误的强信号** —— 足球比赛没有 0 分钟进球（最早也是 1'），所以这个字段不是"数据可能 = 0"，而是"手维护时丢了"
- **加进 `_is_incomplete` 而不另起新函数** —— 保持 API 表面简单（一个 `merge_from_api` 一个判定），不暴露 `_is_malformed` 这种细粒度概念
- **不区分"数量不够"和"格式错误"** —— 对调用方来说都是"这个 entry 不可信，用 API 覆盖"，区分只对日志有意义（`logging.warning` 文案以后可以细化）

### 已知限制

- 其它格式错误（如 "A.Isak" vs "A. Isak" 拼写不一致、`stoppage` 字段填错等）仍然抓不到。可以通过更严的 `validate_entry` 解决，但当前不是用户痛点
- `_is_incomplete` 现在混了两种语义（count mismatch + minute=0），将来如果再加判定函数签名可能需要改
