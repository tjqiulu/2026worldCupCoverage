## 关键发现

直接调 `https://worldcup26.ir/get/games` 验证数据源：

```json
{
  "home_team_name_en": "Belgium",
  "away_team_name_en": "Egypt",
  "home_scorers": "{\"محمد هانی 66'\"}",
  "away_scorers": "{\"امام آشور 19'\"}"
}
```

全量排查 15 场已结束比赛：
- 14 场用 Latin 字母（部分带重音，如 Krejčí、Gyökeres、Saibari）
- **1 场（Belgium vs Egypt）纯阿拉伯文**

且**两个进球者都是埃及 Al Ahly 球员**（Mohamed Hany + Imam Ashour），被错分配到不同队——这是 worldcup26.ir 录入错误，不是 transliteration 缺失。

## 决策日志

### 1. 选方案 D（暂入 backlog，等观察）
**问题**: 怎么修？
**候选**:
- (A) 手维护覆盖 + 新增 `_protected_from_api` 标志位 —— 立刻可做，但 66' 主队进球者真实身份未知（比利时国脚无 Mohamed/Imam）
- (B) API 拉取层做 Arabic → Latin transliteration —— 修不到根因（transliteration 后还是"比利时人 Mohamed Hany"，UI 看起来对但数据错）
- (C) UI 层 fallback tooltip —— 治标
- (D) 暂不修，入 P2 backlog，等 6/16 比赛日结束再回看是不是单例（**采用**）
**决定**: D
**理由**: 根因是"数据脏"不是"缺转写"，方案 A/B 都在掩盖脏数据；让数据自曝其短（多观察 1 天）再做决策更稳

### 2. 不动 `data/details.json` 的 `323786f24db4` 条目
**问题**: 现在 `data/details.json` 里这条 entry 是阿拉伯文，要不要先手改？
**决定**: 不改
**理由**: 
- 改了之后下次 refresh 又会被 API 覆盖回去（Plan 017 的合并策略对完整条目不保护）
- 要保护就需要新增 `_protected_from_api` 标志位，那又回到方案 A
- 不如先记 backlog，等数据自爆（如果 6/16 还有其它 Arabic-only 比赛就清楚了）
